FAQ: Is it Possible to Get the Size of Each Record in Dataverse?

The Direct Answer: No, it is not possible to get the precise, all-inclusive storage size of each individual record in Dataverse through standard administration tools.

Why Isn't This a Feature?

The platform is designed to track storage at a higher, aggregated level (by table) for performance and architectural reasons.

  1. Aggregated by Design: Dataverse tracks storage consumption at the table level. The underlying database architecture is complex, and a single Dataverse record can have its data spread across multiple internal tables, making a per-record calculation inefficient.
  2. Complex Storage Model: A single logical record can consume all three types of storage:
    • Database Storage: For structured data like text and numbers.
    • File Storage (Azure Blob): For attachments in notes or emails.
    • Log Storage: For the record's audit history.
    Because these components are physically separate, there is no single "record size" metric available.

The Practical Solution: A Top-Down Approach

While you can't get the size of a single record, you can still effectively find and manage the largest consumers of your storage by starting high and drilling down.

Step 1: Identify the Largest Tables

Use the Power Platform Admin Center (PPAC) storage report to find which tables use the most storage. Navigate to Resources > Capacity, select your environment, and review the consumption tabs.

Step 2: Analyze Data Within Those Tables

Once you identify a problematic table (e.g., Annotation for File storage), you can investigate its data.

Example: Finding Large Attachments

The Annotation (Note) table has a filesize column that stores the attachment size in bytes. You can query this table using tools like Power Automate or the XrmToolBox to sort notes by filesize and find the largest ones.

Summary

Your Goal The Recommended Action
Get the size of a specific, single record. Not Possible.
Find what is consuming the most storage. 1. Check the PPAC report to find the largest tables.
2. Query the data within those tables to find the largest items (e.g., sort notes by filesize).
